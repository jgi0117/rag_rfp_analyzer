# A100 Server vs OpenAI API Cost Break-Even

RAG 기반 RFP 문서 QA에서 A100 GPU 서버를 월 단위로 임대하는 방식과 OpenAI API를 사용하는 방식의 비용 손익분기점을 정리한다.

## 핵심 결론

현재 코드 기준 실제 RAG generation 입력은 질문만이 아니라 다음 전체 프롬프트다.

```text
QA instruction
+ retrieved context
+ user question
```

따라서 비용 계산은 질문 토큰만 보면 안 되고, 검색된 문맥까지 포함한 `qa_prompt` 입력 토큰을 기준으로 봐야 한다.

현재 측정값과 가정은 다음과 같다.

| 항목 | 값 |
|---|---:|
| A100 월 고정비 | `$2,682.57` |
| 평균 입력 토큰 | `1,882.02 tokens/question` |
| 출력 토큰 | `512 tokens/question` |
| 입력 단가 | `$0.75 / 1M tokens` |
| 출력 단가 | `$4.50 / 1M tokens` |

질문 1개당 OpenAI API 비용은 다음과 같다.

```text
입력 비용 = 1,882.02 / 1,000,000 * $0.75 = $0.001411515
출력 비용 = 512 / 1,000,000 * $4.50 = $0.002304
총 비용 = $0.003715515 / question
```

즉, 현재 RAG prompt와 출력 512토큰 기준으로는:

```text
질문 1개당 약 $0.00372
원화 기준 질문 1개당 약 5.02원
```

## 손익분기점

```text
A100 월 비용 / 질문 1개당 OpenAI 비용
= $2,682.57 / $0.003715515
= 약 721,991 questions / month
```

따라서 월 질문 수가 약 `72.2만 건`을 넘으면 A100 서버 임대 방식이 OpenAI API보다 비용상 유리해지기 시작한다.

## 코드에서 중요한 부분

`experiments/baseline_generation.py`에서는 Vector DB에서 검색한 문맥을 모두 합쳐 프롬프트에 넣는다.

```python
joined_contexts = "\n\n".join(row["retrieved_contexts"])

qa_prompt = (
    "당신은 제안서 분석 전문가입니다. 주어진 문맥에만 철저히 기반하여 질문에 답하세요.\n"
    "문맥에 없는 내용이거나 확인 불가능한 정보라면 솔직하게 '문맥상 확인할 수 없습니다'라고 답하세요.\n\n"
    f"[문맥]:\n{joined_contexts}\n\n[질문]:\n{question}"
)
```

비용을 크게 만드는 부분은 고정 프롬프트 문구가 아니라 `joined_contexts`다. 현재 설정에서는 `retrieval.top_k = 4`라 질문마다 최대 4개 chunk가 문맥으로 들어간다.

또한 현재 generation 설정은 다음과 같다.

```yaml
generation:
  provider: huggingface
  model: Qwen/Qwen3-8B
  temperature: 0.0
  max_new_tokens: 512
```

따라서 OpenAI API로 대체할 때도 출력 비용은 실제 운영 상한인 `512 tokens` 기준으로 보는 것이 더 보수적이다.

## 비용 최적화 포인트

비용을 낮추려면 입력 문맥 길이와 출력 길이를 줄이는 쪽이 가장 직접적이다.

- `retrieval.top_k`를 줄이면 입력 토큰이 감소한다.
- chunk 크기와 overlap을 조정하면 retrieved context 길이가 줄 수 있다.
- reranker나 score threshold를 적용하면 불필요한 context 유입을 줄일 수 있다.
- 답변 형식을 제한하거나 `max_new_tokens`를 낮추면 출력 비용이 감소한다.
- 반복되는 prefix나 문서 context가 많다면 cached input 활용 가능성도 검토할 만하다.

## 최종 판단

현재 실험, MVP, baseline 평가 단계처럼 질문 수가 적은 구간에서는 OpenAI API가 비용적으로 유리하다. 하지만 목표가 오픈소스 LLM의 답변 품질을 OpenAI API 수준까지 끌어올리는 것이라면 판단 기준이 달라진다.

두 모델의 품질이 비슷해진다는 전제에서는 A100 서버 임대 방식이 다음 장점을 가진다.

- 월 질문 수가 약 `72.2만 건`을 넘으면 OpenAI API보다 경제적으로 유리해진다.
- 트래픽이 커질수록 질문당 한계 비용이 낮아져 절약 금액이 커진다.
- 고정비 기반이므로 답변 품질을 위해 출력 토큰을 더 넉넉하게 사용할 수 있다.
- 긴 retrieved context와 긴 답변이 필요한 RAG 서비스일수록 API 대비 비용 이점이 커질 수 있다.
- 모델, 프롬프트, retrieval 전략을 직접 통제하면서 서비스 요구사항에 맞게 최적화할 수 있다.

다만 손익분기점은 단순히 월 질문 수만으로 판단하면 안 된다. 현재 손익분기점인 약 `72.2만 건/월`을 A100 1대가 질문을 순차 처리한다고 가정하면, 30일 기준 end-to-end 평균 처리 시간은 질문당 약 `3.59초` 이하여야 한다.

```text
30일 = 2,592,000초
2,592,000초 / 721,991건 = 약 3.59초/질문
```

여기서 end-to-end 처리 시간은 retrieval, prompt 구성, 모델 추론, 답변 생성, 후처리까지 포함한 시간이다. 실제 운영에서는 일부 질문의 context가 더 길거나, 답변 생성이 더 오래 걸리거나, 특정 시간대에 트래픽이 몰릴 수 있다. 따라서 경제적 이점을 안정적으로 가져가려면 단일 순차 처리 가정보다는 batching, 동시 요청 처리, vLLM/TGI/TensorRT-LLM 같은 serving engine 기반 병렬 처리 전략을 함께 적용해야 한다.

결론적으로 단기 실험과 초기 검증은 OpenAI API가 적합하다. 반면 오픈소스 LLM 품질이 충분히 올라오고, 월 처리량이 손익분기점을 안정적으로 넘으며, 병렬 서빙을 통해 실제 처리량을 확보할 수 있다면 A100 서버 기반 직접 서빙은 비용과 운영 자율성 측면에서 더 좋은 선택이 될 수 있다. 반대로 처리량이 부족하거나 병렬 처리 효율을 확보하지 못한다면 OpenAI API가 여전히 더 경제적일 수 있다.
